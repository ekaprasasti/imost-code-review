B Architecture & Patterns — Vertical Slice: Approval MoM

FE 🟡 Acceptable
  • 🟢Pola atomic design konsisten: components/atoms · molecules · organisms · ui.
  • 🟢Setiap fitur (MoM, ESG, Konstruksi, Operasi, dst) memiliki sub-folder api/store/types/const tersendiri.
  • 🟠Auth state via Zustand dengan persist; namun lib/auth.ts kosong (1 baris) — kemungkinan referensi yatim.
  • 🟠Banyak console.log tertinggal di kode produksi (termasuk log payload login: '<<<< USER', 'data id jomok').
  • 🟠: any tersebar pada API layer fitur (app/moms/api/moms.ts, app/[slug]/items/...).
  • 🟢Validasi form via zod + @hookform/resolvers terpasang.
  • 🟡Penamaan konsisten kebab-case untuk route segments; ada beberapa folder PascalCase di operasi/Komersial, operasi/Operasi — inkonsistensi minor.
BE 🔴 Red Flag
  • 🔴src/controllers/main/mom.main.ts = 5.132 baris — fat controller ekstrem mencakup logika MoM, issue, task, approval, attachment, komentar, notifikasi.
  • 🔴Versi handler hidup berdampingan: createMomController & createMomControllerV2; updateMomControllerV7; getAllMomGroupedByStatusV3; getMomIssuesControllerV3; updateMomIssueControllerV4; route lama dikomentar tetapi controller belum dihapus.
  • 🔴Setiap controller membuat new PrismaClient() sendiri (di mom.main.ts, dashboard.main.ts, legal.main.ts, esg.main.ts, photo.main.ts, new.main.ts, project.admin.ts, dashboard.admin.ts) — multi-instance Prisma, risiko kebocoran connection pool meskipun sudah ada libs/prisma.lib.
  • 🟠Logika bisnis (cron + axios + write Prisma) ditanam inline di src/index.ts (lihat baris 210–261).
  • 🟢Error handler tersentralisasi melalui class/error.ts + handleError.
  • 🟠Validasi Joi hanya pada 4 file (auth · esg · folder · main); banyak endpoint MoM & admin tanpa validator terpusat — sebagian malah validate inline di controller.
  • 🟡Pola route memang konsisten menerapkan auth.authenticateToken; tetapi per-feature authorization hampir tidak pernah dipasang (lihat src/routes/mom.route.ts — 28 endpoint tanpa authorizeFeature).
  • 🔴Duplikasi kode masif terdeteksi melalui versi V1–V7 — refactor approval flow akan menyentuh banyak titik.
Slice trace — Approval MoM: Route POST /api/v1/mom/approval → middleware authenticateToken saja → handler approvalMomControllerV2 di file 5K-baris. Tidak ada layer service/repository; query Prisma, sanitasi, notifikasi, dan respons digabung dalam controller. Authorization per-feature tidak diterapkan untuk approval — siapa pun pemegang token dapat mengakses endpoint approval selama payload menunjuk mom_id yang sah.

C Database & Prisma

BE 🟠 Attention Required
  • 🔴prisma/migrations/ hanya berisi satu direktori: 20241203194035_hehe. History migrasi tidak dapat dijadikan dasar reproduksi schema; nama migrasi tidak profesional.
  • 🟠schema.prisma sebesar 4.052 baris berisi 70 model & 43 enum. Banyak penambahan kolom baru akan memerlukan migrasi enum (overhead operasional).
  • 🟢Foreign key Prisma @relation(...) banyak terpasang dengan onDelete: SetNull / onUpdate: SetNull; total 70 @@index/@index/@unique terhitung.
  • 🟠480 field String? (sangat nullable) dan 302 kolom soft-delete/timestamp (deleted_at · is_deleted · deletedAt) — soft delete tersebar, tetapi belum standar (penamaan campur).
  • 🟠Sisa artefak framework lama: model failed_jobs dengan uuid · connection · queue · payload · exception · failed_at — pola tabel Laravel queue, mengindikasikan schema awal di-port dari project Laravel.
  • 🟠Sebagian FK pada model legacy dikomentari (mis. cyber_security_regen relasi projects) — referential integrity tidak dijamin database.
  • 🟢Tidak ditemukan $queryRaw/$executeRaw dengan string interpolation — risiko SQL injection rendah.
  • 🟡Penggunaan include/select pada findUserByEmail sudah baik; namun pada query MoM, controller sering mengambil objek besar tanpa select — potensi N+1 perlu diverifikasi pada slice lain.
  • 🟠Banyak tipe domain memakai String? alih-alih enum yang sudah tersedia (mis. critical_path.status, remark) — kontrol nilai lemah.

D Security & RBAC Assessment

FE 🔴 Red Flag
  • 🔴app/api/sign-in/route.ts memuat credential demo hardcoded: test@plnnr.com / demo1234 → mengembalikan token statis sample-token. Endpoint ini dapat di-exploit oleh siapa pun yang tahu URL-nya.
  • 🔴RBAC hanya di sisi klien melalui hook usePermission (membaca localStorage.auth_permissions). Menyembunyikan tombol bukan otorisasi — endpoint backend tetap harus memeriksa, tetapi (lihat BE) belum konsisten.
  • 🟠Token JWT disimpan di localStorage selain cookie — terpapar XSS. Cookie tersetel secure; sameSite=strict (baik), tetapi tanpa HttpOnly (karena diset client-side via Cookies.set).
  • 🟠Hardcode origin gambar ik.imagekit.io + rewrites ke https://api.imost.ptplnnr.com — konfigurasi tidak lewat ENV.
  • 🟡Tidak ditemukan raw HTML/innerHTML berbahaya; PDF render via @react-pdf aman.
BE 🔴 Red Flag
  • 🔴authorizeFeatureWithCache (src/middlewares/authenticate.middleware.ts) selalu memanggil next() dan men-set cache 'allowed' tanpa pernah menanyakan database — otorisasi efektif dimatikan pada endpoint apa pun yang memakainya (mis. POST /auth/who-am-i).
  • 🔴authorizeFeature versi non-cache tidak memanggil next() pada jalur sukses (hanya menangani !permission → 403); permintaan dengan permission valid akan menggantung.
  • 🔴Endpoint GET /seed di src/index.ts mengeksekusi runSeed() tanpa autentikasi — siapa pun dapat re-seed database produksi.
  • 🔴console.log(hashedPassword) dan console.log(isMatch) di login.auth.ts — kebocoran data otentikasi ke log.
  • 🟢Hashing password memakai Argon2 (type: 2) — selaras praktik modern.
  • 🟢JWT memakai HS512, expiry 7 hari, secret di ENV; helmet, express-rate-limit, dan validasi tipe file (image/jpeg · png · pdf) sudah dipasang.
  • 🟠CORS allowlist hardcoded di kode (lihat index.ts:29) mencampur https://domain1.com placeholder, banyak port localhost, dan domain produksi — risiko salah konfigurasi rilis.
  • 🟠Cron job di index.ts hit URL produksi http://103.196.155.16:3005/fetch-jawa7 via HTTP plain — secrets/endpoint pihak ketiga di-hardcode.
  • 🟢File upload divalidasi via mimetype + nama disanitasi (sanitize-html) sebelum disimpan.
Kombinasi paling berbahaya: FE menyembunyikan kontrol akses di klien, BE menonaktifkan authorizeFeature* secara efektif → seluruh perlindungan per-feature bergantung pada keberadaan token, bukan hak akses pengguna. Setiap user terotentikasi berpotensi mengakses fitur tingkat admin selama frontend memunculkannya.

✓ Rekomendasi & Dampak Estimasi

Aksi Cepat (1–2 minggu)

  • Hapus endpoint /api/sign-in demo di FE dan endpoint /seed di BE.
  • Perbaiki middleware authorizeFeature* agar benar-benar memvalidasi permission.
  • Pasang authorizeFeature(...) pada semua route MoM, Admin, ESG, Legal.
  • Bersihkan console.log sensitif & sembunyikan log debug di production.
  • Audit package.json FE; buang paket palsu & downgrade React ke versi stable.

Stabilisasi (1–2 bulan)

  • Refactor mom.main.ts ke service/repository layer; hapus controller V1..V6 yang sudah usang.
  • Singletonkan Prisma client; tambah lint rule untuk mencegah instansiasi liar.
  • Pisahkan cron / job runner dari index.ts; gunakan BullMQ/agenda atau sidecar.
  • Buat baseline migration profesional & alurkan migrasi via CI/CD.
  • Bangun CI minimal: lint + typecheck + unit + smoke test.

Dampak terhadap estimasi

Posisi risiko keseluruhan dinilai tinggi. Buffer effort yang disarankan pada saat pengambilalihan: +35% s.d. +50% dari baseline scope untuk fase stabilisasi keamanan & refactor controller MoM, di luar pekerjaan fitur baru. Disarankan menjadikan butir Risk Register R-01..R-05 sebagai blocker serahterima.

Rekomendasi singkat (satu paragraf)

Kedua repository memiliki fondasi arsitektur yang cukup (Next.js App Router dengan atomic design; Express + Prisma + Postgres dengan Argon2, JWT, Helmet, Prometheus), namun kebersihan rilis dan disiplin RBAC masih jauh dari siap produksi. Tiga temuan paling kritis — credential demo hardcoded di FE, middleware otorisasi yang efektif no-op di BE, dan endpoint /seed terbuka — harus diperbaiki sebelum serahterima dianggap selesai. Jejak fake-editor.sh juga menunjukkan perlu dilakukan klarifikasi atas riwayat kepemilikan kode. Catatan penting: asesmen ini adalah 30-minute vertical-slice skim sesuai panduan checklist, bukan audit menyeluruh — pemeriksaan mendalam disarankan terutama pada modul ESG, Legal, dan Admin yang belum ditelusuri seluruhnya.

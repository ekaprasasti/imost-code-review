7 Daftar Masalah Ditemukan

  • 🔴 Cron scraping Cirata tidak aktif — body kosong, hanya console.log. Data realtime Cirata bergantung pada sistem eksternal di luar repo.
  • 🔴 Auth dimatikan — auth.authenticateToken dikomentar pada route /overview/:slug. Siapa pun yang tahu URL bisa akses data project.
  • 🟠 Hardcoded URL produksi — scraper Jawa-7 panggil http://103.196.155.16:3005 via HTTP plain, tanpa env var, tanpa auth.
  • 🟠 MongoDB credentials di source code — admin:yoursecurepassword hardcoded di libs/mongo.lib.ts:4.
  • 🟠 Logic per-slug yang tidak scalable — switch(slug) hanya support pltu-jawa-7 & plts-cirata. Project baru akan return 0 untuk data realtime tanpa error.
  • 🟠 Magic number — monthlyConverted = 326277482.45 / 1000 di dashboard.main.ts:528 tanpa penjelasan, diterapkan ke semua PLTU/PLTS.
  • 🟠 power_plants[0] tanpa null-check — project tanpa power_plant akan crash dengan TypeError: Cannot read properties of undefined.
  • 🟡 Dummy data di response JSON — produced:"1250 MWh", output_power:"185 MWp", revenue:"330 Million", energy_equivalent:"203.05 MWh" — hardcoded string, bukan dari DB.
  • 🟡 Dead select — project_metrics_history.anually_contract_energy di-fetch tapi tidak masuk response.
  • 🟡 Tabel scrap tanpa FK — scrap_data_cirata tidak punya project_id / relasi ke projects; coupling cuma lewat hardcoded slug.
  • 🟡 Cron interval inkonsisten — komentar bilang "every 5 minutes" tapi schedule */1 (tiap menit) untuk Cirata, dan */6 untuk job dashboard padahal log bilang "every 10 minutes".
  • 🟡 Bug popup hardcoded — overview-api.ts:26 selalu fetch pltu-jawa-7 walaupun user buka project lain.

8 Rekomendasi

Jangka Pendek

  • Aktifkan kembali auth.authenticateToken di route /overview/:slug.
  • Pindahkan URL scraper & MongoDB ke env var.
  • Implementasi/restore cron fetch-cirata mengikuti pola Jawa-7, atau dokumentasikan service eksternal yang mengisi tabel.
  • Tambah null-check power_plants?.[0] untuk hindari crash.
  • Hapus dummy strings dari response atau ganti dengan null.

Jangka Menengah

  • Refactor switch(slug) → generik via scrap_data_* berelasi FK ke projects.
  • Pindahkan dummy/magic number ke kolom DB (project_metrics_history) atau env config.
  • Pisahkan business logic ke service layer; controller jangan akses Prisma langsung.
  • Pisahkan tabel scrap_data_cirata ke service "scraping" sendiri (microservice / TimescaleDB).
  • Hapus select field yang tidak dipakai (project_metrics_history).
